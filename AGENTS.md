# AGENTS.md

Guidance for coding agents working in this repository.

## Project Overview

`ash_cheatsheet` is a small Go web app for storing and viewing cheatsheet cards by section. It uses:

- Go 1.23 module `ash_cheatsheet`
- Standard-library `net/http` route patterns such as `GET /sheet/{section}`
- Server-rendered `html/template` templates in `templates/*.htm`
- HTMX for form posts, preview updates, edits, and deletes
- Local static assets in `static/`
- SQLite database at `data/db.sqlite`, initialized from embedded Goose migrations in `internal/db/migrations`
- A custom PEG-based markup parser under `internal/grammar`, consumed by `internal/render`

## Important Commands

- Run the app: `make run`
- Run the app directly: `go run cmd/app/main.go`
- Run tests: `go test ./...`
- Run the grammar sandbox: `make sandbox`
- Regenerate parser from grammar: `make peg`

The app listens on `127.0.0.1:8080`.

## Repository Layout

- `cmd/app/main.go` wires templates, database, services, and HTTP handlers.
- `cmd/sandbox/main.go` is an experimental parser/rendering sandbox.
- `internal/cards` contains the card service and repository interface.
- `internal/db` contains the SQLite client and embedded migrations.
- `internal/entities` contains shared domain structs.
- `internal/errs` contains service-level sentinel errors.
- `internal/grammar` contains the PEG grammar, generated parser, and AST conversion.
- `internal/render` converts custom markup into HTML and has the main tests.
- `internal/handlers/*` contains one package per HTTP handler.
- `templates/*.htm` are full pages or HTMX response fragments.
- `static/` contains Bulma, HTMX, Font Awesome font data, and app CSS.

## Architecture Notes

- `main.go` parses all templates with `template.ParseGlob("templates/*.htm")`, then handlers select named templates with `tmpl.Lookup(...)`.
- The service layer (`internal/cards`) depends on a small `Repository` interface. Keep business validation there when it is card-related.
- Handlers parse route variables via `request.PathValue(...)`, which depends on Go 1.22+ `net/http` pattern routing.
- Form handlers currently read `request.Body` and parse it with `url.ParseQuery`; match that style unless doing a broader cleanup.
- Rendered card descriptions are generated by `internal/render.Render`, then passed to templates as `template.HTML` in `getsection`. Ensure rendered output remains escaped/sanitized before expanding markup features.
- `internal/grammar/grammar.go` is generated from `internal/grammar/grammar.peg`. Edit the `.peg` file first, then run `make peg`.
- `internal/render/archive` appears to be preserved old rendering code. Do not route new behavior through it unless explicitly reviving it.

## Database Notes

- Migrations are embedded from `internal/db/migrations/*.sql`.
- Runtime data lives in `data/`, which is gitignored. Do not commit local SQLite files.
- Add schema changes as new migration files instead of editing migrations that may already have been applied locally.
- Check column names carefully when touching database code. The migration currently defines `last_updated`, while the Go DB layer queries and inserts `updated_at`; resolve that mismatch deliberately if you work on persistence.

## Frontend Notes

- UI is mostly Bulma plus custom `static/global.css`.
- HTMX attributes drive preview, create, edit, and delete interactions.
- Templates contain Russian user-facing text in add/edit result flows. Preserve existing language unless the task is explicitly about localization.
- Static files are served by `internal/handlers/static`, which only allows `.css`, `.woff2`, and `.js`.

## Testing Expectations

- Run `go test ./...` before finishing Go changes.
- Add or update focused tests when changing `internal/render`, `internal/grammar`, service validation, or handler behavior.
- Parser/rendering changes should include examples covering escaping, inline code, bold text, multi-line text, and code blocks where applicable.
- For generated parser changes, include both the `.peg` source change and regenerated `grammar.go`.

## Code Style

- Follow existing package boundaries and keep handlers small.
- Use `gofmt` on changed Go files.
- Prefer returning/wrapping errors in service and storage code. Current handlers often call `logger.Fatal`; avoid spreading that pattern into lower layers.
- Keep generated files mechanically generated. Avoid manual edits to `internal/grammar/grammar.go`.
- Avoid unrelated cleanup or formatting churn in templates and CSS.

## Working Tree Caution

This repo may contain local data, editor files, or in-progress UI changes. Check `git status --short` before editing, and do not revert changes you did not make.
